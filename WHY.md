# 이거 만든 이유

개멍청한 토요일의 낮이 시작.

---

## 배경

9명짜리 OSS 조직에서 GitHub 웹훅 이벤트를 Discord로 쏴주는 봇을 만드는 중이었다. 이슈, PR, CI, 릴리즈 전부 채널별로 라우팅되는 그런 봇. 마지막 퍼즐 조각은 `repository.created` 이벤트 핸들러였다. 새 레포 생기면 봇이 알아서 Discord 포럼 채널에 스레드를 만들어주는 기능.

이걸 end-to-end로 테스트하려면 실제로 레포를 만들어서 이벤트를 쏴야 했다. 테스트 레포 생성 → 봇 동작 확인 → 테스트 레포 삭제. 이 세 단계.

망할 이유가 없었다 ...

## 클릭

봇이 테스트 레포를 감지 못 했다. 어깨 한 번 으쓱하고 테스트 레포 Settings 페이지로 들어가서 지우고 다시 해보려고 했다. Danger Zone까지 스크롤. 'Delete this repository'. 이름 타이핑. 빨간 버튼 클릭.

다음에 로드된 페이지는… 레포 페이지가 아니었다. 조직 overview 페이지. 더 정확히는 **띠껍게 생긴 404.**

나는 내내 **조직** Settings 페이지에 있었던 거였다. 내가 누른 "Delete this repository"는 실제로 "Delete this organization"이었고, 내가 타이핑한 확인 텍스트는 레포 이름이 아니라 조직 이름이었다. 자연스럽게 복사 및 붙여넣기로 레포의 이름을 썼는데, 그거는 레포가 아니라 조직의 이름이였다.

9명의 레포, 이슈, PR, 디스커션, 릴리즈, 웹훅, 시크릿, CI 히스토리. 한 번의 클릭으로 사라졋다

## 복구

알고 보니 GitHub는 삭제된 조직을 복구해준다. 30일 이내에 서포트 티켓 열면. 무료 계정은 영업일 기준 3~7일 걸린다. 그 며칠 동안 할 수 없는 것들:

- 해당 조직의 어떤 레포에도 push
- 이슈나 PR 보기
- URL로 뭔가 참조하기
- 팀원한테 "그냥 풀 받으면 됨"이 왜 안 되는지 설명하기
- 머릿속에서 그 클릭 장면 안 돌리기

할 수 있는 건: 로컬 git remote를 개인 계정으로 바꿔서 일단 작업 이어가기.  

근데 이거도 어쨌든 손해는 손해여서 딱히 회복은 안 됐다......,,..

## 계속 맴돌던 생각

GitHub에는 "삭제된 레포 복구"가 셀프서비스로 있다. 버튼 한 번 클릭, 15초면 끝.

GitHub에는 "삭제된 조직 복구"가 셀프서비스로 없다. 지원팀에 메일 보내고 일주일 기다려야 한다.

**100배 더 많은 데이터를 날릴 수 있는 기능이, 셀프서비스로 가능한 기능보다 더 약한 가드레일로 보호되어 있다.**

조직 삭제 확인 모달은 레포 삭제 모달이랑 똑같이 생겼다. 복붙도 된다. "이 작업이 멤버 9명, 레포 47개, 이슈 213개에 영향을 줍니다" 같은 요약도 없다. 10초 쿨다운도 없다. 파괴력에 비례하는 마찰이 아예 없다.

나는 UX 디자이너가 아니다. 하지만 이것에 대해선 의견이 있다.

## 그래서 뭘 햇나

차피 내가 직접적으로 그걸 막기는 힘드니까 좀 우회해서 막아보기로 했다.  

for-real은 Chrome 익스텐션이다. GitHub의 "Delete this repository" / "Delete this organization" 버튼 클릭을 감지하고, 현재 페이지가 미리 등록해둔 보호 목록에 있으면 클릭을 가로챈다. 그리고 레포/조직 루트 페이지로 튕겨내면서 모달을 띄운다. 모달에는 랜덤으로 생성된 20자 코드. **직접 타이핑해야 한다. 복붙 안 된다.** 통과하면 5분간 삭제 언락, 진심이면 Settings 다시 들어가서 삭제 진행하면 됨.

반나절 걸렸다. 이건 내가 아니라 GitHub가 기본으로 제공해야 하는 기능이라는 거 안다. 그래도 만든 이유:

1. 복구 기다리는 동안 생산적으로 느껴지는 유일한 일이었다
2. 이 익스텐션이 단 한 명이라도 내가 한 짓을 못 하게 막는다면, 반나절은 충분히 값어치 있다
3. 18:54에 느꼈던 감정을 다시는 느끼고 싶지 않다

## 배운 것

- 파괴적 버튼의 마찰은 파괴력에 비례해야 한다. 레포 삭제 플로우에서 재활용하기 편했던 UI 그대로 쓰면 안 된다
- "로컬에 있어요"는 코드에는 훌륭한 백업이지만 이슈, PR, 디스커션에는 아무 쓸모 없다
- 비슷하게 생긴 페이지 두 개는 언젠가 헷갈린다. 그걸 전제로 설계해야 한다
- 조직 관리한다면 이런 거 하나 깔아둬라. 이거든 다른 거든 직접 만들든. **뇌와 그 버튼 사이에 뭐든 끼워둬라.**
